# stories.md -- modelfoundry (python)

This document breaks the `modelfoundry` project into an ordered sequence of small, independently completable stories grouped into phases. Each story has a checklist of concrete tasks. Stories are organized by phase and reference modules defined in `tech-spec.md`.

Put **`vX.Y.Z` in the story title only when that story ships the package version bump** for that release. Doc-only or polish stories **omit the version from the title** (they share the release with the preceding code story, or use your project’s doc-release policy). **One semver bump per owning story** — extra tasks on the *same* story share that bump; see `project-essentials.md`. Semantic versioning applies to the package. Stories are marked with `[Planned]` initially and changed to `[Done]` when completed.

For a high-level concept (why), see [`concept.md`](concept.md). For requirements and behavior (what), see [`features.md`](features.md). For implementation details (how), see [`tech-spec.md`](tech-spec.md). For project-specific must-know facts, see [`project-essentials.md`](project-essentials.md) (`plan_phase` appends new facts per phase). For the workflow steps tailored to the current mode (cycle steps, approval gates, conventions), see [`docs/project-guide/go.md`](../project-guide/go.md) — re-read it whenever the mode changes or after context compaction.

## Version Cadence

Standard semantic versioning, with these conventions:

- **Every story belongs to a phase.** Bugfix stories included. No orphan stories.
- **Per-story bumping** (when a story owns its own release):
  - Bugfix or trivial change → **patch** (`vX.Y.Z+1`)
  - Feature or improvement → **minor** (`vX.Y+1.0`)
  - Breaking change → **major** (`vX+1.0.0`). Post-1.0 only, and only via the `plan_production_phase` mode, which negotiates with the developer about whether the breakage is substantively user-facing or technically-but-trivially breaking (example: a log-format change is technically breaking, but if logs aren't a core consumer capability, the developer may judge it minor or even patch).
- **Phase-bundling option:** a phase can run unversioned during work and ship a single release/tag at end-of-phase. Stories within the phase carry no version in their title; the phase's last story owns the bump (magnitude determined by the highest-impact change in the bundle).
- **No out-of-order implementation.** Story order in this file is the order of execution. If work order needs to change, **reorganize/renumber here first** — don't skip ahead and create version-number gaps.
- **Pre-1.0:** standard semver applies; version starts at `v0.1.0` (Story A.a).
- **Post-1.0:** every phase must go through `plan_production_phase` (the lighter `plan_phase` is pre-1.0 only). Major bumps only happen through that mode's negotiation step.

This is the authoritative cadence rule. **Do not extrapolate the bump magnitude from `pyproject.toml`'s current version** — re-read this section whenever you're about to assign a version to a story.

### Story H.a: v0.8.4 Fix normalization units mismatch in `DataRefineryDataset` — model trains to chance [Done]

### Story H.d: v0.9.2 Apply lazy augmentations before normalization — fix train/eval skew [Done]

`features.md` CR-16 / QR-1. Cache-invalidating for augmented recipes (materialized output changes); pre-production OR-9, no `schema_version` bump. Owns **v0.9.2** (bugfix patch).

**Gap analysis (verified 2026-06-17 by an end-to-end probe) — why this is a bundle, not one story.** PyTorch already materializes *and* validates through a recipe (the existing `simple_cnn`/`resnet20` baseline ops + `cross_entropy` loss + `adamw`/`sgd`/`adam` optimizer ops). - **Gap A — `validate()` is broken for every sklearn recipe.** The sklearn plugin registers **only** the `mlp_classifier` architecture op (verified: `sklearn -> {'architecture': ['mlp_classifier']}`), but `ModelRecipe` *requires* `Loss` and `Optimizer` blocks, and validator check 3 (`section_ops_registered`, [recipe/validator.py:135](../../src/modelfoundry/recipe/validator.py#L135)) rejects any op the plugin doesn't register → `ops not registered by plugin 'sklearn': [('Loss', 'cross_entropy'), ('Optimizer', 'adamw')]`. - **Gap B — no recipe-driven random/chance baseline.** No `plugin: random` / `Architecture.type: dummy_classifier`; the only chance baseline ([scripts/examples/test_random_classifier.py](../../scripts/examples/test_random_classifier.py)) computes `1/N` with numpy off `mf.data.path`, bypassing the recipe. → **H.f.2** adds the plugin, built validate-complete from the start so it doesn't repeat Gap A.
- **Gap C — the sklearn swap is not a one-block change.** A faithful pytorch→sklearn swap changes `plugin:` + `Architecture:` **and** drops `Optimization:`/`Visualizations:` (sklearn raises `NotImplementedError` for both; the runner gates Optimization on presence ([runner.py:105](../../src/modelfoundry/pipeline/runner.py#L105)) and skips the model summary, so omitting them materializes cleanly). The tutorial must state this honestly — *identical glue + data binding; the recipe changes plugin + architecture and omits the stages the baseline doesn't implement* — not "change one line." Addressed as copy in **H.f.3**.
- **Gap D — baseline coupling + comparison (secondary, out of scope).** The sklearn feature path reuses the PyTorch `DataRefineryDataset` ([plugins/sklearn/data.py](../../src/modelfoundry/plugins/sklearn/data.py)), so the baseline currently needs the `[pytorch]` extra (a documented pre-production trade); and `Evaluation.comparison.baseline_model_id` (FR-12) is a no-op in the sklearn path. The tutorial uses three separate materializations and notes the torch dependency; both deeper fixes are `plan_features` follow-ups.

**Architectural fork (developer's call — flagged at the gate).** Gap A has two fixes: **(A) per-plugin op registration** — sklearn/random register the Loss/Optimizer ops their recipes use (localized; keeps the schema requiring those blocks; lets sklearn's Optimizer block actually drive the MLPClassifier `solver`/`learning_rate_init`); or **(B) core-schema relaxation** — make `Loss`/`Optimizer` optional in `ModelRecipe` (like `Optimization`) so baseline/dummy plugins needn't declare them. The bundle assumes **Option A** (smaller, pre-production-appropriate, makes the recipe meaningful); Option B is a larger core change (recipe model + validator + every plugin) better suited to `plan_features`.

**Bundle order & versions:** H.f.1 (sklearn validate fix + minimal recipes, **patch v0.9.3** — restores the documented flow) → H.f.2 (random plugin, **minor v0.10.0**) → H.f.3 (README tutorial, no bump) → H.f.4 (capacity-vs-budget study, *static stopping*, no bump) → H.f.5 (sophisticated regime — cosine LR + early stopping, no bump) → H.f.6 (10x data on MPS — the crossover confirmed, no bump) → H.f.7 (full 50k dataset, same split proportions, no bump) → H.f.8 (**debug**: regime vs residual bug — version conditional).
